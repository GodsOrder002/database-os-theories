你现在要思考的，其实不是“哪个更快”这么粗的维度，而是：**你把变化留在 JSON（协议路线）之后，数据库怎么把“常用检索路径”变快**。MySQL 和 MongoDB 的差别，主要就落在这点上。

---

## MySQL 的 JSON：能用，但它更像“把 JSON 当二等公民”

MySQL 的 JSON 是原生类型（会校验合法性、内部是二进制存储，读写比把 JSON 当纯文本强）。([MySQL][1])

但关键限制是：**JSON 列本身不能像普通列一样直接建索引**。常见解法是把 JSON 里某个路径抽出来做 generated column 或 functional index，再对那个表达式建索引。([MySQL][2])
另外 MySQL 8 也有对 JSON 数组的 multi-valued index（适合做“标签数组 contains X”那类检索）。([Medium][3])

所以 MySQL 的性能画像是：

* **你知道会经常查哪些 key/path → 可以做到很快**（因为你能“把协议的热点”固化成索引入口）
* **你想随时查任意 key/path → 很容易退化成函数扫描**（JSON_EXTRACT/JSON_CONTAINS 这类函数如果没命中索引路径，就会吃 CPU）([MySQL][2])

这和你说的“协议路线 + UI 动态加字段”天然有张力：MySQL 不是不能承载动态，但**动态到一定程度，你就必须接受“只有热点能快，长尾靠扫描”**。

---

## MongoDB：JSON（BSON）就是一等公民

MongoDB 的文档模型天生就是为“字段随时长出来”设计的，索引可以直接建在任意字段上，包括嵌套字段、数组字段。([MongoDB][4])
它还有 schema validation（可选），用 JSON Schema 约束集合结构，这对“协议路线但不想完全失控”很友好。([MongoDB][5])
查询/聚合方面，Mongo 的 aggregation pipeline 有专门的优化机制；而你一旦走到需要“管道式解析 + 过滤 + 分组”的场景，它会更顺手。([MongoDB][6])

代价也很清晰：

* 关系型那套“强一致引用/复杂 join”不是它的母语；虽然有 `$lookup`，但它本质上是聚合阶段的 join，复杂度和性能边界要更谨慎管理。([MongoDB][7])
* 当你真的需要大量跨实体、跨维度的关联分析，SQL 的表达力和优化器生态仍然是优势区。

---

## 放到你这类“key_json 标签化协议”的现实结论

你现在想要的结构是：**弱结构 + 可检索标签（key_json）**，且允许策略者自己定义检索键。

* 如果你预期检索模式会很快收敛成少数几类（比如：按 creator、按 status、按某几个固定 tag、按某几个固定维度过滤）
  → **MySQL JSON 是可控的**：把这些热点维度固化成索引入口，其余长尾靠 AI 检索/召回在应用层兜底。MySQL 的“强项”是：你愿意把一部分协议“翻译成索引”。([MySQL][2])

* 如果你预期检索维度会长期发散、而且经常要查“策略者临时发明的字段/嵌套结构/数组条件”
  → **MongoDB 更贴**：它对“任意字段随时可索引”这件事更自然，协议路线的动态成本更低。([MongoDB][4])

一句话：
**MySQL 的 JSON 适合“协议可变，但热点可固化”；MongoDB 适合“协议长期发散，但仍要靠索引吃饭”。**
